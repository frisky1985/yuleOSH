## 风险分级

| 等级 | 含义 | 处理时限 |
|------|------|----------|
| 🔴 P0 | 阻塞性风险，必须解决后再推进 | 立即 |
| 🟡 P1 | 高影响风险，需有缓解方案 | 本阶段内 |
| 🔵 P2 | 中影响风险，需有监控和降级方案 | 跟踪 |
| ⚪ P3 | 低影响风险，记录在案 | 持续观察 |

## 一、数据一致性风险

### R1: 图谱与代码实际状态不一致 🟡 P1

**描述**: 知识图谱通过 CI 增量构建，但 CI 可能被跳过、失败、或不完整。此时图谱状态落后于代码仓库实际状态。

**影响**:
- 追溯查询结果不准确
- 变更影响分析遗漏或误报
- PR 评论中的信息陈旧

**缓解**:
1. **隔夜全量重建** — 定时任务（每日 UTC 0:00）执行图一致性检查，对比代码仓库/测试结果的 head 状态，自动修复漂移
2. **快照版本化** — 每次构建生成 `kg_snapshots`，支持比较任意两个快照的 diff，可手动回滚到已知正确的快照
3. **Staleness 标记** — 如果超过 N 次构建未更新（`stale_build_count`），标记边为 `stale: true` 并在查询时告警
4. **CI 强制排期** — 核心管道的增量构建设为 `required`（非 `optional`），跳过即阻断

### R2: 启发式 `implements` 推导误报 🟡 P1

**描述**: 基于测试覆盖反向推导 `implements` 边（TestFunction covers ReqR，TestFunction verifies CodeFuncF → 推导 F implements R），可能存在误报：
- 测试覆盖了 A，A 调用了 B，推导 B implements ReqR（实际上 B 与 ReqR 无关）
- 测试断言的粒度不足以证明代码确实实现了该需求

**影响**:
- PR 评论中列出不相关的需求/测试
- Merge gate 误阻挡
- 开发者失去对图谱的信任

**缓解**:
1. **推导三级置信度**（KG-12）：
   - `explicit` — 代码注释中的 `// implements SHALL-001`（高可信）
   - `derived` — 测试反向推导且无冲突（中可信）
   - `heuristic` — 基于命名模式/文件结构猜测（低可信）
2. **查询默认过滤** — 默认只返回 `explicit` 和 `derived` 级别，`heuristic` 需要显式参数 `--include-heuristic`
3. **AI 审核辅助** — 对于 `derived` 推导，在首次生成后由 AI 审核 `{Requirment.Statement} ↔ {CodeFunction.Signature}` 对，人工确认后升级为 `explicit`
4. **衰减机制** — 如果 `derived` 边连续 N 次 CI 构建未被人工确认→自动降为 `heuristic`

### R3: 跨模块引用完整性 🔵 P2

**描述**: TKG 的 `relates_to` 边引用知识管理模块的 KB/FMEA 条目。如果知识模块条目被软删除、版本变更、或状态变更，TKG 中的引用可能不一致。

**影响**: 图谱中存在指向不存在或已废弃的知识条目的悬挂边。

**缓解**:
1. **复用 CROSS-03** — 知识管理模块已定义引用完整性规则，TKG 的 `relates_to` 边走同一套逻辑
2. **定期清理** — 定时任务检查悬挂引用，自动标记 `stale: true`
3. **级联通知** — 当知识条目状态变更为 `deprecated` 或软删除时，自动通知 TKG 更新关联边

## 二、性能风险

### R4: 递归 CTE 查询性能下降 🔵 P2

**描述**: PostgreSQL 递归 CTE 在深度≥5 跳、大规模图中性能下降显著。根据性能预估，50K 节点以上 5 跳查询可能超过 500ms。

**影响**: API 响应超时、CI 报告生成缓慢、用户体验差。

**缓解**:
1. **深度限制**（KG-20）— 默认递归深度上限 5，可在查询时调整
2. **查询缓存** — 热点查询（如 `impact(src/foo.py)`）结果缓存 60s
3. **GIN 索引** — `properties` JSONB 字段建 GIN 索引，加速属性过滤
4. **异步报告生成** — PR 影响分析报告异步生成，通过 webhook 更新 PR 评论
5. **阈值触发的 Neo4j** — 当总节点 > 50K 时，提示可选用 Neo4j 后端；>100K 时默认推荐

### R5: 增量构建在大型 PR 上超时 🟡 P1

**描述**: 大型 PR（50+ 文件变更）上，增量构建可能需要解析大量文件、提取函数、分析依赖，导致构建超时。

**影响**:
- CI 步骤被终止
- PR 不生成影响分析报告
- 图谱不完整

**缓解**:
1. **超时降级** — 增量构建设置超时（默认 60s），超时后记录变更文件列表但不执行深度解析，生成"部分构建"快照（标记 `partial: true`）
2. **隔夜全量补全** — 超时降级后，由隔夜全量任务补全缺失的深度分析
3. **分片处理** — 将大变更按模块分片处理，每个分片独立构建
4. **文件过滤** — 排除非代码文件（文档、图片、配置文件）的图更新

### R6: CI 构建总时长增加 🔵 P2

**描述**: 增加 TKG 增量构建步骤（估算 10–30s）将延长 CI 管道总时长，影响开发者体验。

**影响**: 开发抵触、CI 步骤被禁用。

**缓解**:
1. **异步执行** — TKG 构建步骤与 CI 主流程异步，不阻塞测试结果返回
2. **配置化** — 开发者可通过 `.yuleosh.yaml` 关闭此步骤（`enabled: false`）
3. **仅差异构建** — 仅处理 changed_files，不做全量扫描
4. **渐进式启停** — 默认 P0 阶段仅在 `on_spec_change` 时触发（避免每次提交都跑），P1 升级为 `always`

## 三、技术风险

### R7: RTM 与知识图谱的双写问题 🔴 P0

**描述**: 引入 TKG 后，需求→测试的追踪信息有两处记录：
- 现有的 `requirement-traceability-matrix.md`（静态文档）
- TKG 的 `kg_nodes` + `kg_edges`（图谱）

如果二者不同步，团队不知道该信哪个。

**决策**: 
**P0–P1 阶段：双写同步。** RTM 保留作为人类可读的静态快照，但实际生成逻辑改为 **从图谱导出**。`requirement-traceability-matrix.md` 由 `yuleosh kg export --format markdown` 自动生成，不再手动维护。

**P2 阶段：淘汰静态 RTM。** 作为默认选项，RTM 不再作为 CI 产物的一部分"写死"，改为"按需导出"。

**缓解**: bootstrap 阶段从 RTM 导入后，立即标记"该 RTM 已被图谱取代，后续变更请通过图谱查询"。

### R8: PostgreSQL 递归 CTE 在特定查询上的边界表现 🔵 P2

**描述**: 递归 CTE 的特性约束：
- 不支持无限递归（默认限制 100，可达 1000 但影响性能）
- 不支持路径聚合（如"查找所有路径"而非"查找最短路径"）
- CTE 中 `UNION ALL` 可能导致重复遍历

**影响**: 部分图分析功能受限，无法完全替代 Neo4j。

**缓解**:
1. 明确 CTE 限制的使用场景
2. 聚合分析（扇入/扇出）通过物化视图预计算
3. 复杂路径分析（如多条件最短路径）推迟到 P3 Neo4j 可选项

### R9: 现有 yuleOSH 面向 C 代码，注释格式限制 🔵 P2

**描述**: 显式追溯声明（如 `// covers SHALL-001`）依赖代码注释格式，而嵌入式 C 代码可能使用不同的注释风格或编码规范。

**缓解**:
1. 支持多种注释格式：`// SHALL-001 covers`、`/* @trace SHALL-001 */`、`// [TKG] implements RS-001`
2. 解析器配置化，支持 `.yuleosh.yaml` 中定义注释模式
3. 无注释时的启发式推导作为 fallback

## 四、运营风险

### R10: 团队认知成本 🔵 P2

**描述**: TKG 是新增概念，团队需要理解图模型、查询方式、CI 集成。如果认知成本过高，可能被搁置不用。

**影响**: 投入开发后无人使用。

**缓解**:
1. **渐进式复杂度** — P0 最简单（已有 RTM 直接导入），P1/P2 逐步解锁
2. **CLI 而非 API** — 初期仅提供 CLI 命令（`yuleosh kg query`），降低使用门槛
3. **PR 评论为默认入口** — 开发者不直接使用 TKG，但 PR 评论自动包含影响分析
4. **文档与 demo** — 提供 5 分钟上手教程

### R11: 知识图谱的维护退化风险 ⚪ P3

**描述**: 任何自动构建系统随时间都可能出现偏差，如果团队不信任图谱数据，会退化到手动维护的 RTM。

**缓解**:
1. **定期一致性审计** — 每月运行图对代码仓库的一致性检查，生成不符合率报告
2. **构建成功率仪表盘** — 展示 TKG 构建成功率、修复次数、staleness 标记数量
3. **人工确认反馈机制** — 开发者可以在 PR 评论中点击"确认"或"报告错误"按钮，反馈进入 LL 条目

## 决策日志

| 日期 | 决策 | 理由 |
|------|------|------|
| 2026-07-14 | P0 选 PostgreSQL 边表而非 Neo4j | 零新增基础设施，当前规模（~400 节点）CTE 足够 |
| 2026-07-14 | 双写同步而非 RTM 退役 | P0 阶段兼容过渡，RTM 改为从图谱导出 |
| 2026-07-14 | 启发式推导默认不参与门禁 | 避免误报导致开发者反感 |
| 2026-07-14 | 增量构建异步化 | 不阻塞 CI 主流程，降低 P0 部署摩擦 |
