## 分阶段方案

### Phase T1 — 快速修复（低难度，高回报）

| 项 | 工作量 | 说明 |
|----|--------|------|
| 修复覆盖率工具 bug | 1天 | 升级 coverage 版本或改用纯 Python  tracer |
| 审核并修复 Wave 5 测试文件 | 0.5天 | 6个文件需要验证 schema 正确性 |
| 建立集成测试框架 | 2天 | 为 LLM/硬件模块创建 mock fixture 库 |

### Phase T2 — 中难度攻坚

| 项 | 工作量 | 说明 |
|----|--------|------|
| pipeline/step_handlers/ 测试 | 3天 | 7个大文件，创建 mock Agent/LM 环境 |
| alm/ 适配器测试 | 2天 | dSPACE/Vector/Jira/Polarion mock 适配器 |
| autosar/ 测试 | 1天 | stubgen 等模块 |

### Phase T3 — 高难度（依赖硬件/外部）

| 项 | 工作量 | 说明 |
|----|--------|------|
| hardware/ + cross/ + sil/ 测试 | 4天 | 需要 HIL/SIL 框架或 mock |
| CI 门禁 fail-under 60 | 与 T2/T3 并行 | 随覆盖率提升逐步提高 |

---

## 优先级矩阵

| 象限 | 高价值 | 低价值 |
|------|--------|--------|
| **低成本** | T1: 修复覆盖率工具 + 子 agent 测试 | T1: 文档补充 (Phase T1) |
| **高成本** | T2: step_handlers 测试 + alm 测试 | T3: hardware HIL 测试 |

**推荐执行顺序：** T1 全做 → T2 做 step_handlers + alm → T3 视硬件资源决定

## 评审建议

1. 接受当前 50% 覆盖率作为基线
2. 将 T1 作为立即执行项（低风险高回报）
3. T2 在生产功能开发中并行推进
4. T3 在有 HIL/硬件资源时启动
