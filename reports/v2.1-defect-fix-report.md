## DF-001: alloca/VLA 运行时检测增强

### 发现问题
`review_bsp.py` 已包含全面的 BSP 静态检查（pin-mux 冲突、时钟树 PLL、HAL 初始化顺序、DMA 配置），但完全缺失**运行时分配**检测：
- 未检测 `alloca()` 使用（MISRA C:2023 Rule-21-5 禁止）
- 未检测 VLA（变长数组）（MISRA C:2023 Rule-18-7 禁止）
- 未检测动态内存分配（MISRA C:2023 Dir 4.12 禁止）

这在嵌入式 BSP 代码审查中是关键缺口，因为：
- BSP 代码运行在受限栈空间（~1-4 KB）的 MCU 上
- 中断处理程序和 RTOS 任务栈更小
- alloca/VLA 触发栈溢出时无错误返回，直接静默破坏内存

### 修复内容

新增 4 个检测函数：

| 函数 | 检测目标 | 严重度 |
|:-----|:---------|:------:|
| `_check_alloca_usage()` | `alloca()` / `strdupa` / `strndupa` | 🔴 critical |
| `_check_vla_usage()` | VLA 模式（`type var[n]` where n 不是常量） | 🟡 major |
| `_check_dynamic_allocation()` | `malloc`/`calloc`/`realloc`/`free` + NULL-check | 🟡 major/minor |
| `_check_runtime_allocation_integrity()` | 跨文件聚合运行上述 3 个检查 | — |

### 验收
- 检测到 `alloca()` 时输出 **critical** finding
- 检测到 VLA 时输出 **major** finding（附示例代码片段）
- 检测到动态分配时区分少量（minor）和大量（major）
- 检测到动态分配但未做 NULL-check 时增加一条 major finding

## DF-002: test_c_unit.py `$$` PID 展开 bug

### 发现问题
```python
# 原始代码（bug）
result = subprocess.run(
    ["gcc", "-o", "/tmp/c_test_runner_$$"]
    ...
)
```
`$$` 在 subprocess.run(list) 中不会被 shell 展开。程序始终写入字面量文件 `/tmp/c_test_runner_$$`，重复运行会：
1. 覆盖同一个文件（多进程竞争条件）
2. 如果文件被锁则失败

### 修复内容
```python
# 修复后
tmp_runner = os.path.join(
    tempfile.gettempdir(),
    f"c_test_runner_{os.getpid()}_{id(session)}"
)
result = subprocess.run(
    ["gcc", "-o", tmp_runner]
    ...
)
# 运行后清理
os.unlink(tmp_runner)
```

### 验收
- 每次运行使用唯一文件名（PID + session id）
- 运行后自动清理临时二进制文件
- 无文件竞争条件

## DF-003: gcov_coverage.py fail_under 门禁 (CL2-E03)

### 发现问题
`gcov_coverage.py` 采集 C 覆盖率数据并生成报告，但完全未实现 CL2-E03 要求的 **fail_under 阈值检查**。
- 即使行覆盖率 = 0%，脚本也报告 success
- `stages.py` 虽已有 `run_c_coverage_check()`，但需要先有 `gcov_coverage.py` 的独立门禁能力

### 修复内容
1. **`generate_c_coverage_report()`** 新增参数：
   - `fail_under: Optional[float]` — 行覆盖率阈值
   - `fail_under_branch: Optional[float]` — 分支覆盖率阈值
2. **JSON 输出新增字段**：
   - `gate_passed: bool | None` — 门禁是否通过
   - `gate_details: list[dict]` — 每个指标的通过/失败详情
3. **CLI 新增参数**：
   - `--fail-under <N>` — 行覆盖率门禁（例：--fail-under=60）
   - `--fail-under-branch <N>` — 分支覆盖率门禁（例：--fail-under-branch=50）
4. **CLI 退出码**：门禁失败时 `exit 1`，通过时 `exit 0`

### 使用示例
```bash
# 生成报告 + 检查门禁
python -m yuleosh.ci.gcov_coverage \
  --build-dir=build \
  --src-dir=src \
  --fail-under=60 \
  --fail-under-branch=50

# 输出示例:
#   C coverage report: /path/.yuleosh/reports/c-coverage.json
#   ✅ PASS: line_rate=72.3% >= 60%
#   ❌ FAIL: branch_rate=45.0% >= 50%
#   ❌ Coverage gate: FAILED — threshold(s) not met
#   exit code: 1
```

## DF-004: coverage_trend.py 回归告警 (CL2-E04)

### 发现问题
CL2-E04 step 4 明确要求：
> "设置回归告警阈值：单次下降 > 5% 触发 pipeline warning（非 blocking）"

`coverage_trend.py` 已有 `record_coverage()` 记录和 `show_coverage_trend()` 展示，但缺少**回归告警**功能。

### 修复内容
新增 `check_coverage_regression(project_dir, ...)` 函数：

| 参数 | 默认值 | 说明 |
|:-----|:------:|:-----|
| `line_drop_threshold` | 5.0 | 行覆盖率下降百分点的告警阈值 |
| `branch_drop_threshold` | 5.0 | 分支覆盖率下降百分点的告警阈值 |
| `window` | 3 | 滑动窗口大小（用于计算基线平均值） |

**函数输出**：
- `regression: bool` — 是否检测到回归
- `alerts: list[dict]` — 告警详情（类型、严重度、当前值、基线值、下降幅度）
- `latest: dict` — 最新覆盖条目
- `baseline_avg: dict` — 基线平均值
- `status: str` — "passed" 或 "warning"

### 验收
- 覆盖数据不足时输出 `no_data` / `insufficient_data` 告警
- 最新 C 行覆盖率低于滑动平均超过 5pp 时输出 `c_line_regression` 告警
- 支持 C 和 Python 的行/分支覆盖率回归检测
- 不抛出异常（返回结构化结果）

## DF-005: sync_check.py YAML Schema 验证 (CL2-E05/E06)

### 发现问题
`s ync_check.py` 已有基础 doc→code tracking 功能（E06），但缺少：
- **CL2-E05**: YAML Schema 文档验证 — 验证文档必需字段完整性
- **CL2-E06**: 关键模块差异化门禁 — 区分 error/warning

### 修复内容

#### E05: YAML Schema 验证
新增 `DOC_YAML_SCHEMAS` 常量定义 3 种文档 Schema：
| 文档类型 | 必需字段 |
|:---------|:---------|
| architecture | `module_name`, `version`, `last_updated`, `code_path` |
| interface | `interface_name`, `parameters`, `return_type`, `changelog` |
| requirement | `requirement_id`, `description`, `status`, `code_module` |

新增 `validate_doc_yaml_schema(project_dir)` 函数：
- 扫描匹配文件模式
- 解析 YAML → 验证必需字段
- 返回 error（缺失字段）/ info（字段完整）

#### E06: 增强门禁
新增 `run_sync_check_gate(project_dir)` 函数：
- 组合 E05（Schema 验证）+ E06（代码→文档 tracking）
- 统一状态判定（failed / warning / passed）
- 输出全面报告

CLI 新增 `--enhanced` 参数：
```bash
python -m yuleosh.ci.sync_check --enhanced
```

## 覆盖率门禁状态确认 (Defect 4)

### 当前状态

| 门禁类型 | 位置 | 配置 | CI 中是否生效 |
|:---------|:-----|:----|:------------:|
| Python 行覆盖率 | `pytest.ini` → `--cov-fail-under=80` | 80% | ✅ GitHub Actions 直接运行 |
| Python 行覆盖率 | `.github/workflows/ci.yml` | 80% | ✅ |
| C 行覆盖率 | `.yuleosh/ci-config.yaml` → `c_fail_under: 70` | 70% | ⚠️ 已实现但在 GitHub Actions CI 中被"skipped" |
| C 覆盖率门禁 | `stages.py` → `run_c_coverage_check()` | 70% | ⚠️ 需 LLM API key 才能完整运行 |
| gcov_coverage.py | `--fail-under` CLI 参数（本次新增） | 按需 | ✅ 新增 |

### 结论
- Python 覆盖率门禁：**已完整生效**
- C 覆盖率门禁（stages.py）：**代码已实现**，但 GitHub Actions 中 `yuleosh.ci.run 1` 因需要 LLM API key 被跳过
- gcov_coverage.py：**本次修复新增独立门禁**，可直接在 CI 中通过 `--fail-under` 参数使用

### 建议
1. 在 GitHub Actions 中添加一步独立的 C 覆盖率检查：
   ```yaml
   - name: C Coverage Gate
     run: |
       python -m yuleosh.ci.gcov_coverage \
         --build-dir=build --src-dir=src --fail-under=60
   ```
2. 该步骤不需要 LLM API key，可在 GitHub Actions 中正常运行
