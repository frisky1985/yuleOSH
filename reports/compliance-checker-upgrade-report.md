## 1️⃣ 改造内容

### 1.1 新增方法（3 个）

| 方法 | 可见性 | 功能 |
|:-----|:------|:-----|
| `_get_kg_store()` | private | 尝试从 `yuleosh.knowledge_graph` 工厂获取 store；KG 不可用时返回 `None` |
| `_check_with_kg(check_item, kg_store)` | private | 根据 check_item 关键字匹配 KG 查询并返回 `True`/`False`/`None（降级信号）` |
| `_get_kg_stats(kg_store)` | private | 收集图谱统计信息：节点/边数、implements/covers/validates 边数、per-layer 覆盖、CI snapshot 数 |

### 1.2 改造方法（2 个）

| 方法 | 改造点 |
|:-----|:-------|
| `run()` | 生命周期内获取一次 KG store → `_get_kg_stats()` → 注入 `report["kg_data"]`；向下传递 `kg_store` 给每个 `_check_bp()` |
| `_check_bp()` | 每个 check_item 先尝试 `_check_with_kg()`，返回非 None 则使用 KG 结果（标记 `[KG]`），返回 None 则降级到原有文件检查逻辑 |
| `generate_report_markdown()` | KG 数据存在时，在 Summary 后插入 `## KG Data (Real Traceability)` 信息块 |

### 1.3 不变的部分

- `ComplianceChecker` 构造函数签名不变（`project_dir` + `template_path`）
- `run_and_save()` 输出文件格式向后兼容
- 所有原有文件检查逻辑不受影响（KG 返回 None 时自动降级）

## 2️⃣ KG 检查逻辑映射

| check_item 关键字 | KG 查询函数 | 条件 | 降级策略 |
|:-----------------|:------------|:-----|:---------|
| `trace` / `bidirectional` | `get_graph_stats()` → `edges_by_type.implements > 0` | implements 边存在 | → 文件存在性检查 |
| `unit test` / `unit verification` | `get_aspice_coverage()` → `unit.total_covers > 0` | unit 层 covers 边存在 | → 文件存在性检查 |
| `integration` / `confirm` / `validat` | `get_confirmation_trace()` → len > 0 | validates 边存在 | → 文件存在性检查 |
| `snapshot` / `ci result` | `list_snapshots()` → len > 0 | CI snapshot 存在 | → 文件存在性检查 |
| 其他 | — | — | 纯文件检查 |

**安全设计**: 所有 KG 操作包裹在 `try/except Exception` 中，异常时返回 `None` 触发降级（不抛不中断）。
