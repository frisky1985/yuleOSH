1. **CL2 就绪是唯一优先级** — 所有非 CL2 项（模块拆分、注释、SHOULD 测试）全部降权
2. **关键路径优先** — E01→E02→E03→E10→E13 和 E08→E09→E13 两条关键路径上优先

4. **老陈审查 = 有成本** — 每次审查 = 老陈约 1~2 天工时（按日计费），不宜过度频繁

## 2. CL2 审查验收标准

### 2.1 CL2 综合就绪度门槛

基于 MISRA 验收矩阵 §20.x 的 41 项评估和 §20.x 判定标准：

| 条件 | 门槛 | 判定 |
|:-----|:----:|:----:|
| **PA 2.1 TM 就绪度** | ≥90%（≥13/14） | **硬性门禁** |
| **PA 2.2 MP+RI 就绪度** | ≥85%（≥19/22） | **硬性门禁** |
| **混合项（G-50 CLI + Dry Run）** | ≥80%（≥4/5） | **硬性门禁** |
| **CL2 综合就绪度** | **≥90%（≥37/41）** | **硬性门禁** |
| Dry Run 审计 | 无 major/critical 发现 | **硬性门禁** |
| 证据包可一键生成 | 全部 6 子目录非空 | **硬性门禁** |

**老陈认可 ⭐ = CL2 综合就绪度 ≥ 90% + Dry Run 无 major 发现**

### 3.2 为什么这样安排

**Wave 2 后邀请老陈的理由**：
1. CL2 核心证据（G-47~G-50）刚就位，老陈的方向性反馈能避免 Wave 3 做偏
2. 73% 就绪度是"有东西可看但明显还有缺口"的状态，老陈的反馈性价比最高
3. 如果等 Wave 3 全部做完再请老陈，万一核心设计方向错了，返工成本巨大

**不每 wave 都邀请的理由**：
1. **烧钱**: 老陈（前博世资深架构师）按日计费，1 次审查≈1~2 天工时。6 月 18 日已经烧了五轮，不能无限烧
2. **团队节奏**: 小克是主力开发者，每次审查准备也得花时间（至少 0.5 天整理证据），过度频繁的审查会打乱开发节奏
3. **边际收益递减**: 第一阶段审查差距大（58→70→76→85→90），第二阶段到 CL2 90% 差距更细，审查的边际收益递减，应该把老陈的精力集中在关键节点
