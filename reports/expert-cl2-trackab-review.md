## 五、发现的问题 🔴

### 🔴 P0 — 要立即处理

1. **Python 覆盖率 9%** —— 11/11 个关键模块覆盖率 < 25%
   - 基准：CL2 要求工具代码本身经得起审计
   - 现状：21,408 语句中 18,817 未覆盖
   - 建议：别贪 C 覆盖了，先把 yuleOSH 自己的 Python 测试做到 60%+

2. **审查追溯 0/184（0%）** —— 这是最让我失望的
   - 184 个 SHALL 需求没有一个关联到审查记录
   - 门禁 H1 里最弱的一环
   - 跟证据包 reviews/ 为空是同一件事的两个表现

3. **CI 流水线 FAILED** —— 跟邀请函内容矛盾
   - yaml-validation 失败（`misra-c2023-3.2.description: expected str, got NoneType`）
   - unit-tests 因覆盖率不足失败
   - 邀请函写"412+ tests all pass"，实际是 FAILED
   - 这个矛盾会被审计师抓住不放

### 🔴 P1 — 需尽快处理

4. **证据包 reviews/ 目录仍然为空** —— 这是我上次就说了的问题
   - 上次说"EvidenceCollector 碎片化"，Track B 说修了
   - 但证据包生成后 reviews/ 还是空的
   - 7/57 artifact 状态是 failed
   - 这叫"修了一半"

5. **MISRA Required 违规 14 条超阈值 5**
   - KPI 唯一一个挂掉的指标
   - 虽然违规在 benchmark 文件中，但阈值设了就要守

6. **misra-check 没找到真实 C 代码**
   - CI 报告：`No C/C++ source files found`
   - 但你们有 benchmark C 代码
   - 这说明 ci-config.yaml 配置或规则解析有问题
   - yaml-validation 失败也是相关的

7. **过程稳定性 KPI 最近 14 天无数据**
   - 最后一条是 2026-06-08
   - 说明要么 pipeline 不跑了，要么数据采集断了

8. **缺陷逃逸数据太少（仅 3 条）**
   - CL2 审计师至少要看 20-30 条缺陷数据才有统计意义
   - 3 条不足以支撑 H7 构建元数据
