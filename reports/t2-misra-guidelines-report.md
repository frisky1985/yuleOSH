### 规则覆盖分析

**YAML 标记 vs 实际能力对比**:

| 来源 | 数量 | 说明 |
|:-----|:----:|:------|
| YAML auto_checkable=True（标记自动） | 178 | 但其中 ~35 条 cppcheck 覆盖率 <70% |
| YAML auto_checkable=False（标记手工） | 15 | 全部确认为纯人工审查（+1 条 mcu-esp32-2 可部分自动化） |
| 本次实际纳入人工审查指引 | 50 | 15条(YAML False) + 35条(YAML True但需人工兜底) |

### 偏差说明

在 `misra-rules.yaml` 中，178 条规则被标记为 `auto_checkable=True`，但实际 cppcheck 对其中约 35 条规则的检测覆盖率不足：

1. **语义层次规则**（如 Rule 1.2 未定义行为、Rule 8.13 const 限定）：cppcheck 只能检测最明显的语法违规，无法理解代码的设计意图
2. **跨文件规则**（如 Rule 17.3 间接递归）：cppcheck 受限于单个翻译单元分析
3. **流敏感规则**（如 Dir 4.1 运行时边界、Rule 18.6 动态索引越界）：cppcheck 的值流分析能力有限
4. **条件编译规则**（如 Rule 2.2 条件编译死代码、Rule 14.3 宏常量控制表达式）：cppcheck 对宏展开后的语义理解不足
5. **硬件相关规则**（如 mcu-* 中断保护和 MMIO 地址）：cppcheck 完全无硬件感知

## 4. 审查建议

### 4.1 自动化的下一步

以下规则可通过 LLM Agent 审查提高覆盖率：

| 优先级 | Rule ID | 自动化方案 | 估工 |
|:------:|:--------|:-----------|:----:|
| P1 | Rule 9.1 | Agent 路径分析：检查条件分支中变量是否全路径初始化 | 1天 |
| P1 | Rule 17.7 | Agent 检查所有函数调用的返回值是否被使用 | 0.5天 |
| P1 | Rule 21.3 | Agent 扫描 malloc/free 配对和 OOM 检查 | 0.5天 |
| P2 | Rule 2.2 | Agent 搜索 `#if 0` / 调试宏死代码 | 0.5天 |
| P2 | Rule 20.7 | Agent 展开类函数宏检查参数多重求值 | 1天 |

### 4.2 人工审查调度建议

| 审查模式 | 频率 | 适用场景 |
|:---------|:-----|:---------|
| 全量审查（50 条） | 每次 Release | 新功能模块首次上线 |
| 增量审查 | 每次 MR | 修改的文件变更涉及中等/复杂级规则 |
| 抽检审查（10 条关键） | Nightly CI | 持续监控关键合规指标 |

### 4.3 审查记录模板

建议在每条人工审查代码记录中添加 `// MISRA-REVIEW:` 标记：

```c
/* MISRA-REVIEW: Dir 4.1 — 所有数组访问有边界守卫，通过 */
/* MISRA-REVIEW: Rule 1.1 — __packed 结构体有偏差 DEV-001 */
```
