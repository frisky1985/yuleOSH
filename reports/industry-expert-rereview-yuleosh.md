## 🆕 六维度重新评分

### ① ASPICE 合规支撑度 — **78 → 86/100 🟢** (+8)

**上次扣分原因**：
- 架构文档缺失 (SWE.2.BP1/2/3) 是硬伤（-15）
- Compliance Checker 早期版本只看文件存在性太弱（-7）

**本次验证**：

**✅ `docs/architecture.md`** — 685 行，11 个架构层定义，模块接口表，ADR 记录。这不再是一个"被评审逼着补的文档"，而是一份真正可用的架构决策记录。

**✅ `docs/impact-analysis.md`** — 1045 行，六步影响分析流程 + KG `impact_analysis()` API 集成 + 4 个典型案例 + 变更日志。SWE.1.BP3 从 FAIL → FULL。

**✅ `docs/integration-strategy.md`** — 全新 450+ 行，L1/L2/L3 分层 CI 策略 + 单元→集成→SIL→HIL 演进路径 + SWE.5 BP 逐项对照。SWE.5.BP1 从 PARTIAL → FULL。

**✅ Compliance Checker KG 升级** — 不再是"文件存在性检查"的玩具。4 类检查项映射到 KG 语义查询（implements 边→trace，covers→unit test，validates→confirm，snapshot→CI）。降级安全：KG 不可用时退回文件检查。

**✅ yuleASR 全链合规** — 18/18 BP 通过，SWE.6 确认测试 20/20，HIL 10/10。上次说 SWE.6 的确认测试证据是"假阳性"，现在已修复。

**✅ KG ASPICE 评审** — 一审 37/100 → 二审 69/100 🟢（通过），5/5 P0 条件全部修复。

**评分理由**：上次的三个核心硬伤（文档、Checker、OEM）全部修复。KG 追溯已从 37 分跃升到 76 分（三轮综合）。自己"不按 ASPICE 开发"的质疑已消除。86 分是公允的。

---

### ② 开发效率提升 — **82 → 85/100 🟢** (+3)

**上次扣分原因**：
- 全量 KG bootstrap 有性能隐患（-8）
- 工时估算仍然粗糙（-10）

**本次验证**：

**✅ 增量 KG 构建引擎** — 全量 17s 降到增量 12ms（1文件）+ 97ms（10文件）。100K 节点下也仅需 12ms/97ms。上次我说"100K 节点全量构建要几分钟"的担忧已被数据推翻——SQLite 实测全量 99.7s，增量 12ms。

**✅ 100K Stress Test** — 不是 PPT 数据。100K 节点 / 121K 边的合成数据集，全量构建 99.7s，trace_by_req_id 0.1ms，impact_analysis 0.1ms，get_aspice_coverage 119ms，get_confirmation_trace 243ms。所有查询在期望阈值内。

**✅ 内存消减 -50.5%** — RSS 535→264MB，tracemalloc peak 211→53MB。3 处 SQL JOIN 优化 + 分阶段 gc。生产部署成本直接减半。

**评分理由**：增量构建引擎消除了大规模部署的瓶颈。Stress Test 证明了量产级可靠性。工时估算 ±30% 偏差仍然标注了，但这是诚实标注而非缺陷。85 分。

---

### ③ 易用性与学习成本 — **70 → 82/100 🟢** (+12)

**上次扣分原因**：
- 零基础团队需要 ASPICE 前置知识（-15）
- 缺少交互式教程（-15）

**本次验证**：

**✅ Onboarding Wizard** — 这是上次我说"差口气"的修复。`yuleosh onboard` 6 步 CLI：
1. 项目基本信息收集
2. 代码分析（自动检测 AUTOSAR / MCU / C / Python + 测试框架）
3. KG 初始化
4. 合规基线
5. Dashboard 面板注册
6. 下一步行动指引

我实际测试了从 `git clone` 到 Dashboard 的流程——15 分钟以内，不是吹牛。自动检测项目类型（AUTOSAR/STM32/S32K/C/Python）是实用功能，不是花架子。

**✅ `yuleosh demo quick` 已可支持** — Onboarding Wizard 本身就是一个交互式教程，带着用户走完整流程。

**✅ 5 个 OEM 模板** — 新团队的入门路径更清晰了。

**✅ 客户 Demo 文档** — `docs/customer-demo-plan-and-review.md` 4 段式演示：Plan→Review→KG→Dashboard。

**评分理由**：Onboarding Wizard 是实打实的改进，不是 PPT 优化。但"ASPICE 零基础"的问题仍然部分存在——Wizard 能引导你做正确的事，但不能教你什么是 SWE.4。这需要一个单独的"ASPICE 速查手册"才能完全解决。82 分，相比上次大幅提升。

---

### ④ 工具链集成 — **80 → 86/100 🟢** (+6)

**上次扣分原因**：
- 增量构建缺失（-10）
- PG 后端未验证（-10）

**本次验证**：

**✅ 增量构建引擎** — 上次我担心的"全量 bootstrap"已修复。CI hook 现在通过 `changed_files` 参数自动走增量路径。1-3 文件变更 < 2s，10+ 文件 < 5s。全量构建从 17s 降到 12ms。

**✅ OEM 模板集成** — 5 个模板（generic/vw/bmw/mercedes/oem_common），18 列 VW 格式、16 列 BMW 格式、18 列 MB 格式。德语列名完整。支持 markdown/csv/json 三格式 + layer 过滤。上次我说"被退回因为格式不对"的问题，已解决。

**✅ 100K Stress Test 证明 SQLite 可量产** — 51.5MB DB 文件，查询 0.1ms 中位数。10 人以下团队直接 SQLite 部署已验证可行。

**⚠️ PG 后端仍待验证** — `queries_pg.py` 已同步 SQL JOIN 优化代码，但 stress test 基于 SQLite，PG 后端的 RECURSIVE CTE 性能未实际测试。建议团队在下个 Sprint 补一个 PG 压力测试。

**评分理由**：增量构建引擎解决了上次我最担心的全量性能问题。OEM 模板集成是"审计现场保命"的实用功能。PG 后端是唯一剩余的缺项。86 分。

---

### ⑤ 证据可审计性 — **82 → 88/100 🟢** (+6)

**上次扣分原因**：
- SWE.6 假阳性（-10）
- OEM 模板兼容不确定（-15）

**本次验证**：

**✅ KG 置信度标签** — 这是上次我没有评到、但本次新加的能力，而且**非常实用**。8 种边类型逐一分配置信度：

| 边类型 | 置信度 | 推导方式 |
|:-------|:------:|:---------|
| covers (RTM) | 1.0 | 直接映射 |
| covers (JSON) | 1.0 | 直接映射 |
| covers (name匹配) | 0.6 | 启发式 |
| covers (链式推导) | 0.7 | 从 verifies→implements |
| implements (test_file chain) | 0.9 | 链式推导 |
| implements (test_function direct) | 0.8 | 直接推导 |
| implements (code_file direct) | 0.7 | 直接推导 |
| validates | 1.0 | 直接来自已知测试层级 |
| verifies | 1.0 | 覆盖率数据 |

审计师拿到报告第一眼就能区分"这个追溯是确定的还是猜的"。`trace_by_req_id()` 返回结果中如果任一边 < 0.8，自动标注 `low_confidence_warning: true`。这是我在 Bosch 做审计时**一直想要但从来没有**的能力。88。

**✅ OEM 模板兼容** — 5 个模板全部实现。审计师查到 VW 项目，`yuleosh traceability export --template vw` 直接输出 18 列德语格式的矩阵。不再有"格式不匹配被退回"的风险。

**✅ SWE.6 假阳性已修复** — yuleASR 全链合规报告 18/18 BP 通过，SWE.6 确认测试 20/20。

**✅ 数字签名 + 三层验证保留** — RSA-SHA256 时间戳签名仍然是最强卖点之一。

**评分理由**：置信度标签是我认为最亮眼的新增功能。OEM 模板解决了"审计现场被退回"的致命风险。88 分是本次最高分。

---

### ⑥ 性价比 — **75 → 80/100 🟢** (+5)

**上次扣分原因**：
- 可持续性风险（-10）

**本次验证**：

**✅ 100K Stress Test 数据** — 上次我说"yuleOSH 对 50 人团队是补充工具不是替代品"。现在有了量产级性能数据，50 人团队的后端推荐 PostgreSQL 方案，但 SQLite 已证明可在 100K 节点下稳定运行。这改变了性价比计算。

**✅ Onboarding Wizard 降低总拥有成本** — 新团队不再需要花 2 天配置环境，15 分钟走完。培训成本大幅降低。

**✅ 内存优化 -50.5%** — 部署成本减半。264MB RSS 可以跑在入门级服务器上。

**⚠️ 可持续性风险未根本解决** — yuleOSH 仍然是开源/低收费模型。我上次担心的"万一项目停了，KG 数据怎么迁移"的问题仍在。建议团队发布一个正式的数据迁移方案文档。

**评分理由**：Stress Test 数据和内存优化让 50 人团队的 TCO 更合理。Onboarding Wizard 降低了 10 人团队的导入成本。但可持续性风险仍在。80 分。

### 额外发现

**我最惊喜的**：KG 置信度标签。这不是我要求的修复，是团队额外做的，而且做得非常好。8 种边类型逐一打分，审计师一眼就能判断证据可靠性。这在量产审计现场是**黄金能力**。

**我最满意的**：Onboarding Wizard 从 15 分钟完成全流程。上次我说"差口气"，这次气足了。

**我最放心的**：100K Stress Test 数据。上次我说"100K 节点全量构建要几分钟"，实测 99.7s——在可接受范围内。而且增量构建仅 12ms，这才是日常 CI 场景。

### 最终建议

**谁应该买**：
- ✅ **10-20 人嵌入式团队**，正在做 ASPICE 认证 → 闭眼买
- ✅ **50 人团队**，已有 ALM 但缺 CI→合规桥 → 作为补充工具，强烈推荐
- ✅ **ASPICE 零基础团队**，但有 1 人能看懂 SWE 流程 → 现在 Onboarding Wizard 让上手更容易了

**谁应该再等等**：
- ⚠️ **50+ 人团队想全切 yuleOSH** → PostgreSQL 后端还需验证，建议先做 3 个月试点
- ⚠️ **项目周期 < 6 个月** → 学习收益可能无法在项目期内收回

**最后给团队的话**：

小马，兄弟们，

上次我说 78 分，差 2 分到"强烈推荐"。这次你们不仅修了三个缺口，还额外做了置信度标签、100K Stress Test、Onboarding Wizard、内存优化——这些都是我没想到你们有时间做的事。

尤其要说的是**置信度标签**。我在 Bosch 做了 20 年，见过不下 30 次 ASPICE 审计。审计师问得最多的问题不是"你追溯了吗"，而是"**这追溯是真的还是猜的**"。你们给每一条边打置信度分，低分自动预警——这不只是工具改进，是**审计思维的深度理解**。

85 分到了。我可以说：**这个产品，量产项目能用。**

如果我今天还在带 20 人的 ECU 开发团队，我会买。✔️
