## 一、嵌入式审查质量 — 4 个新增 Handler 逐项评审

### 1.1 review_linker.py — 链接脚本审查

**代码质量**: 🟢 良

**覆盖检查项**:
| 检查项 | 状态 | 备注 |
|--------|:----:|------|
| 栈/堆大小定义 (STACK_SIZE, __stack_size 等多格式) | ✅ | 有阈值 1KB/2KB |
| 段定义完整性 (.text/.data/.bss/.noinit) | ✅ | critical/major 分级清晰 |
| 中断向量表对齐 (256-byte for Cortex-M) | ✅ | ALIGN 指令扫描准确 |
| 内存区域范围 (MEMORY regions: RAM/ROM/FLASH/SRAM/CCM) | ✅ | hex 解析 + KB 转换 |
| 脚本发现 (.ld/.lds/.scatter) | ✅ | glob 模式合理，有去重 |

**发现的问题**:
1. **LMA vs VMA 区分缺失** — 对于 ARM Cortex-M XIP (eXecute In Place) 架构，`.data` 段的 Load Memory Address (LMA) 在 Flash，Virtual Memory Address (VMA) 在 RAM。当前代码没有检查 `AT>` 或 `>REGION AT>REGION` 语法，这是链接脚本最常出差的地方。很多新手把 LMA 和 VMA 写到同一个区域，导致启动时 data 复制找不到源地址。
2. **缺少 `.ARM.exidx` / `.ARM.extab` 检查** — ARM 架构的异常处理索引表，如果没有正确放置，`backtrace` 和调试器将无法解析调用栈。对 Cortex-M 项目来说这直接影响调试体验。
3. **Flash 起始地址特化检查太弱** — 只报了 info 级别。对于 STM32 系列，Flash 通常从 `0x08000000` 开始，如果有人配成 `0x08001000`（常见错误——写了 flash 分区偏移但没有相应调整），当前只会输出 info，不会报 major 或 critical。
4. **`.heap` 和 `.stack` 区域重叠检查缺失** — 如果 .heap 和 .stack 被分配到 RAM 的同一个子区域，运行时可能出现两者互相覆盖，导致无法解释的随机崩溃。这个坑在 STM32 开发者中非常常见。

**总结**: 形式有内容，阈值选得合理。但深度停留在"文档级"，没到"实战级"——漏了 LMA/VMA 和 heap/stack 重叠这两个嵌入式开发者最容易踩的坑。一个有经验的嵌入式工程师写的检查还会多这两项。

### 1.2 review_startup.py — 启动代码审查

**代码质量**: 🟢 良

**覆盖检查项**:
| 检查项 | 状态 | 备注 |
|--------|:----:|------|
| Reset_Handler 存在 | ✅ | 支持 ASM/C 多格式 |
| __initial_sp 栈指针初始化 | ✅ | LDR SP / MSR MSP 覆盖 |
| .bss 清零循环 | ✅ | sbss/ebss/zerobss 等多别名 |
| .data 复制 (ROM→RAM) | ✅ | __etext / sdata / memcpy 覆盖 |
| 时钟配置 (SystemInit / RCC / PLL) | ✅ | 多模式匹配 |
| main() 调用 | ✅ | BL / B / CALL / __libc_init_array |
| 中断状态 (CPSID/CPSIE) | ✅ | 最佳实践说明 |

**发现的问题**:
1. **🔴 FPU 初始化检查缺失** — 优化计划 P0-03 的 checklist 里明列了"FPU 使能（SCB->CPACR）— 对于 Cortex-M4/M7/M33"，但实际 handler 代码里完全没有。对于浮点运算的 MCU 项目，不使能 FPU 会导致 HardFault。这是一个 **计划写了但代码没做** 的典型例子。
2. **SystemInit 调用时序检查太弱** — 当前只匹配了 SystemInit 符号存在，没有检查它在 main() 之前被调用。在 startup 文件中，SystemInit 的调用顺序应该在 BSS 清零之后、main() 之前。如果有人在 main() 里才调 SystemInit，系统可能以错误的外设时钟先跑了几千条指令。
3. **Reset_Handler 的 C 函数签名匹配不完整** — `void Reset_Handler(void)` 这个常见签名的匹配可能不够健壮（正则 `void\s+Reset_Handler\s*\(` 在复杂格式下可能失败）。
4. **Default_Handler / weak symbol 检查缺失** — 优化计划 checklist 要求检查 Default_Handler 是否为 WEAK/ALIAS，以及所有 ISR handler 是否有覆盖。实际代码没做。未覆盖的中断会跳转到无限循环中的 Default_Handler，这是个运行时陷阱。

**总结**: 核心检查点大多覆盖了，但缺了最关键的 FPU 初始化。优化计划写得很全面的 checklist 没有被完全实现。这件事本身是个 process gap。

### 1.3 review_rtos.py — RTOS 配置审查

**代码质量**: 🟢 良+

**覆盖检查项**:
| 检查项 | 状态 | 备注 |
|--------|:----:|------|
| configMAX_PRIORITIES | ✅ | 阈值 4/32，有 RAM 用量提示 |
| configMINIMAL_STACK_SIZE | ✅ | 阈值 60/120 words |
| 中断优先级分组 (多个宏名适配) | ✅ | STM32Cube + FreeRTOS 多种 |
| 钩子函数 (idle/tick/tickless) | ✅ | 实际业务建议 |
| Tick rate (configTICK_RATE_HZ) | ✅ | 100/1000 Hz 阈值 |
| 互斥量/信号量/队列配置 | ✅ | 多项检查 |
| 调度器优化 | ✅ | configUSE_PORT_OPTIMISED_TASK_SELECTION |

**发现的问题**:
1. **🔴 `configASSERT` 缺失** — 优化计划 P1-02 明确指出"configASSERT 是否定义（建议 Debug 配置启用）"，但实际代码没有检查。`configASSERT` 是 FreeRTOS 排查问题的第一道防线，它的缺失是 major 级别问题，尤其对于调试阶段。
2. **`configCHECK_FOR_STACK_OVERFLOW` 缺失** — 同样是优化计划 checklist 列出的，但代码没做。当前 handler 只检查了栈 canary（在 review_memory 里），但 FreeRTOS 特有的栈溢出钩子没有检查。
3. **`configUSE_DAEMON_TASK_STARTUP_HOOK` 缺失** — FreeRTOS v10+ 引入的特性，对于定时器服务任务初始化后的钩子函数配置，部分项目依赖这个。
4. **`configGENERATE_RUN_TIME_STATS` 缺失** — 虽然没有在优化计划里明确列出，但对于生产级嵌入式系统，运行时统计信息（任务 CPU 占用率）是性能分析的基础。缺少这个配置意味着系统缺乏运行时可视化的能力。

**总结**: 4个 handler 中覆盖最好的一个，对 FreeRTOS 生态理解到位，宏名适配全面。但 checklist-to-code 的 gap 再次出现——优化计划写的 configASSERT 和 stack overflow 检查没实现。

### 1.4 review_memory.py — 内存安全审查

**代码质量**: 🟢 良（最复杂的一个）

**覆盖检查项**:
| 检查项 | 状态 | 备注 |
|--------|:----:|------|
| 动态内存分配检测 (malloc/calloc/realloc/free) | ✅ | 跳过注释，避免误报 |
| 全局变量大小估算 | ✅ | 类型映射+数组展开 |
| 静态递归检测 (static locals in recursion) | ✅ | 括号深度追踪 |
| 环形缓冲区边界安全 | ✅ | 模运算/读写索引 |
| 栈保护 (canary / MPU) | ✅ | __stack_chk_guard + MPU 配置 |
| LLM 样本代码分析 | ✅ | 采样前10个C + 5个头文件 |

**发现的问题**:
1. **🟡 全局变量大小估算过于粗糙** — 当前方法基于逐行正则匹配，只能识别 `int`、`uint32_t`、`char` 等基本类型。它有四个硬伤：
   - **不支持结构体/联合体类型** — `struct can_msg` 或 `uint32_t my_buffer[256]` 不会被正确匹配到行级正则中
   - **不支持 typedef 类型别名** — `my_type global_var;` 完全被忽略
   - **函数内 static 变量不支持** — `void foo(void) { static int counter; ... }` 不会被计入
   - **复杂初始化跳过** — `= { .field = value }` 这类 C99 设计初始化器可能被错误匹配或不匹配
   - 结果：全局变量总大小估算严重偏低，可能只抓到实际量的 20-30%，"~ 12.3 KB" 这样的输出没有实际参考价值
2. **🔴 `alloca()` 和 VLA 检查缺失** — `alloca()`（栈上动态分配）和 VLA（变长数组）在嵌入式领域是禁忌，因为会打破栈空间的可预测性。当前内存审查完全没有覆盖这两个。
3. **🟡 函数栈帧分析缺失** — 当前只检查了全局变量，但嵌入式内存问题的更大来源是函数栈帧。大数组在局部变量中声明（`uint8_t buf[4096]`）是嵌入式栈溢出的头号原因，但当前代码只检查了 global，没有检查 local array sizes。
4. **🟡 static recursion 的 braces depth 逻辑过简** — 简单计数 `{`/`}` 会导致头文件中的结构体定义干扰函数边界判断。对于嵌套结构体或条件编译 `#ifdef` 内的函数，检测精度会大幅下降。

**总结**: review_memory 是四个中最复杂、代码最长的，但也是四个中**实际功能最弱的**——因为内存安全分析本质上需要编译器级别的 AST 支持，靠正则匹配是做不到的。这个 handler 最好的情况是一个"低精度告警器"，最坏情况是给了一个误导性的 "~12.3 KB" 数字让开发者产生虚假安全感。建议后续使用 `-fanalyzer` 或实际编译后的 map 文件分析替代。

```
        形式有内容?  深度到位?  LLM+静态双轮?  遗漏项
review_linker    ✅       ⚠️          ✅          LMA/VMA, heap/stack重叠
review_startup   ✅       ⚠️          ✅          FPU初始化, Default_Handler

```

**总体判断**: **形式大于内容吗？—— 不完全是，但有倾向。**

四个 handler 的代码质量高于我的预期——不是简单的 "call LLM and dump result"，都做了实打实的静态模式检查（600-900 行代码，含正则匹配和业务逻辑）。这与纯粹的"包装性审查"有天壤之别。

但深度确实不够。如果你问"一个干了 10 年嵌入式的工程师遇到的最大坑是什么"，LMA/VMA、FPU 初始化、configASSERT、alloca、heap/stack 重叠——这些才是真正的"嵌入式特色"。当前检查覆盖了"表层嵌入式知识"（栈大小、段定义、RTOS 配置项），但没有覆盖"深层嵌入式知识"（启动到 main 之前的完整时序、内存安全的第一性原理、链接器与运行时环境的交互）。

**关键问题**: 优化计划里 checklist 写得很好（FPU 使能、configASSERT、Default_Handler weak），但代码实现没跟上。**计划写得比代码好**，这是一个 process gap——说明 handler 开发时没有把优化计划的 checklist 作为验收标准来测试。

### 7.1 印象最深的三句话（本轮新发现）

1. **"计划写得比代码好"** — 优化计划中 FPU 检查、configASSERT、Default_Handler weak 都有 checklist，但 handler 代码没有实现。代码开发没有拿着 checklist 逐条打勾。
