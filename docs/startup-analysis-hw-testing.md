# 嵌入式硬件连接测试方案 — S.U.P.E.R 启动分析

> 基于 yuleOSH v0.3.0 现有能力
> 日期: 2026-06-08

---

## S — Situation (现状)

### 已有的

| 能力 | 状态 | 文件 |
|:-----|:----:|:-----|
| Cross-compilation (ARM) | ✅ | Dockerfile.cross + Makefile + CI L2 |
| Cross-compilation (RISC-V) | ✅ | 同上 |
| CI L2 交叉编译阶段 | ✅ | src/ci/run.py: `cross_compile()` |
| Docker 容器化工具链 | ✅ | Dockerfile.cross |

### 缺失的

| 能力 | 状态 | 影响 |
|:-----|:----:|:-----|
| **QEMU 仿真** | ❌ | 无法在 CI 中运行交叉编译产物 |
| **固件刷写** | ❌ | 无法部署到真实硬件 |
| **HIL 测试框架** | ❌ | 无硬件在环测试能力 |
| **串口/日志采集** | ❌ | 无法读取硬件测试输出 |
| **硬件测试报告** | ❌ | 无硬件测试结果集成 |
| **多目标板管理** | ❌ | 无板型配置系统 |
| **测试夹具管理** | ❌ | 无法管理测试环境 |

## U — Understanding (理解)

### 用户故事

```
作为一名嵌入式固件工程师，
我希望在 yuleOSH CI 中：
  1. 自动编译固件 → 刷写到开发板 → 运行测试 → 收集结果
  2. 在 CI L2 中用 QEMU 仿真运行交叉编译产物
  3. 连接真实硬件（STM32 / RISC-V）做 HIL 测试
  4. 测试结果自动生成报告，纳入合规证据包
```

### 分层目标

```
Layer 0: Cross-compile   → ✅ 已有 (Docker + Makefile)
Layer 1: QEMU Emulation  → 新增 (软件仿真，无硬件依赖)
Layer 2: Flash & Run     → 新增 (刷写到真实硬件)
Layer 3: HIL Test        → 新增 (带夹具的自动化硬件测试)
```

## P — Problem Definition (问题定义)

### P1: 编译产物无法在 CI 中验证

```
现状: 交叉编译 hello.c → hello-arm.elf ✅
问题: 没人知道这个 .elf 跑起来对不对
目标: QEMU 仿真执行 .elf，验证基础功能
```

### P2: 没有硬件部署通道

```
现状: .elf 文件留在 build/ 目录里
问题: 无法自动刷到开发板测试
目标: 支持 OpenOCD/pyOCD 刷写 STM32 等目标板
```

### P3: 测试结果无反馈

```
现状: 硬件跑完无人知晓结果
问题: 无法断言测试 PASS/FAIL
目标: 硬件测试结果纳入 CI 状态
```

---

## E — Exploration (探索方案)

### 方案选型

| 层 | 方案 | 工具 | 复杂度 | 优先级 |
|:---|:-----|:-----|:------:|:------:|
| **QEMU** | ARM/RISC-V 仿真 | `qemu-system-arm` / `qemu-system-riscv64` | 🟢低 | **P0** |
| **Flash** | 固件刷写 | `openocd` / `pyocd` + JTAG/SWD | 🟡中 | **P1** |
| **Serial** | 串口日志采集 | `pyserial` + 日志解析器 | 🟡中 | **P1** |
| **HIL** | 硬件在环框架 | `pytest` + 夹具管理 | 🔴高 | **P2** |
