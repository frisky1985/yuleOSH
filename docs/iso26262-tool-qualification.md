## 2. 工具分类 (TCL)

### 2.1 TCL 分类依据

ISO 26262-8 §11.4.3 定义两个工具置信等级：

| TCL | 定义 | 示例 |
|:----|:-----|:-----|
| **TCL1** | 工具不会引入或未能检测到安全相关故障 | 纯文本编辑器 |
| **TCL2** | 工具可能未能检测到安全相关故障 | 静态分析器 |

### 2.2 cppcheck MISRA 分类

| 评估维度 | 结果 | 依据 |
|:---------|:-----|:------|
| 已知误报 (False Positive) 率 | 20-30% | 社区报告及内部积累数据 |
| 已知漏报 (False Negative) 率 | 中等 | MISRA 169 条规则中 ~120 条可自动检查 |
| 工具崩溃/异常 | 极低 | cppcheck 成熟稳定，未见工具崩溃导致漏检 |
| 错误引入风险 | 低 | 静态分析器不修改源代码 |

**评估结论**: cppcheck 归类为 **TCL2**（工具可能未能检测到安全相关故障）。

**理由**: 
- cppcheck MISRA addon 仅覆盖约 120/169 条规则（~71%），约 49 条规则需要人工或 AI 审查补充
- 已知误报率 20-30%，可能掩盖真实违规或产生噪声
- 对于复杂语义（如控制流、指针别名、并发），cppcheck 存在漏报

## 5. 偏差与异常应对

### 5.1 工具升级策略

| 触发条件 | 动作 | 负责人 |
|:---------|:-----|:-------|
| cppcheck 主版本升级 | 重新 qualification 评估 | CI 团队 |
| MISRA addon 版本更新 | 回归验证测试基准 | CI 团队 |
| 发现新漏报模式 | 更新漏报清单并补充审查规则 | 架构师 |

**升级流程**:
1. 在独立分支上升级工具版本
2. 运行测试基准验证 (test-bench/)
3. 比较检测结果的差异 (diff)
4. 量化准确率/召回率变化
5. 更新 qualification 文档
6. 审批后合并

### 5.2 误报处理流程

```
发现误报
  │
  ├─ 确认 → 加入 suppress_rules（按文件/规则）
  │
  ├─ 不确定 → AI/人工审查确认
  │
  └─ 漏报 → 补充至 "已知漏报模式" 清单
             并增加相应 AI 审查规则
```

**suppress 配置示例** (ci-config.yaml):
```yaml
misra:
  suppress_rules:
    - "misra-c2023-10.1:src/legacy/*.c"  # 继承代码误报
    - "misra-c2023-17.7:test/*.c"         # 测试代码豁免
```

### 5.3 异常上报

检测到以下异常时应上报并停止 CI：

1. cppcheck 崩溃或非零退出码（非误报导致）
2. MISRA addon 加载失败
3. 输出格式异常（无法解析）
4. 趋势数据显示违规数异常激增（>3σ 偏差）

### 5.4 定期复核

| 频率 | 内容 | 输出 |
|:-----|:-----|:-----|
| 每周 | 趋势报告审查 | 趋势摘要 + 异常标记 |
| 每月 | 误报/漏报分析 | 更新误报率统计 |
| 每季度 | 工具 qualification 复审 | 更新本文档 |
| 版本发布 | 全量 qualification | qualification 报告 V&V |
