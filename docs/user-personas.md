## 画像 1：质量经理（合规证据 · 追溯）

### 人物画像

| 维度 | 描述 |
|:-----|:------|
| **姓名示例** | 王工，某 Tier 2 汽车电子公司质量经理 |
| **团队规模** | 带领 5-10 人质量团队，对接 20-50 人开发团队 |
| **日常工作** | 维护 ASPICE 过程证据、准备客户审计、管理不合格项闭环 |
| **最大痛点** | 每次客户审计前要花 **1-2 周** 手工整理追溯矩阵和证据包，Excel 版本混乱、漏项频发 |
| **预算权限** | ✅ 有独立质量预算（年度 ¥50-200K 工具预算） |
| **决策风格** | 审慎保守——需要看到同类企业的成功案例 |

### 价值主张

> **"审计准备从 2 周变 5 分钟。"**

yuleOSH 为质量经理解决的核心问题：
1. **一键合规证据包** — 追溯矩阵、验收矩阵、审查日志，`yuleosh ev pack` 一键打包下载
2. **ASPICE 过程域逐项追踪** — SWE.1~SWE.6 每个 BP 完成状态一目了然
3. **变更影响分析** — 需求变更自动标注受影响的设计/代码/测试，证据包增量更新
4. **审计应答就绪** — 审计员问"这个需求的测试覆盖率够不够"，5 秒内给出答案

### 典型场景

> 周一上午 10:00，收到整车厂通知：**"下周三 ASPICE 现场审计"**。王工打开 yuleOSH，运行 `yuleOSH ev check` 检查合规差距 → 发现 SWE.4 缺 2 个 BP 证据 → 调取审查日志补充 → 运行 `yuleOSH ev pack` 生成完整证据包 ZIP。**总耗时 15 分钟。**

## 画像 2：架构师（集成 · 适配器）

### 人物画像

| 维度 | 描述 |
|:-----|:------|
| **姓名示例** | 刘博，嵌入式系统架构师（10年+ 经验） |
| **团队规模** | 负责 20-30 人开发团队的架构决策 |
| **日常工作** | 工具链选型、HIL/SIL 架构设计、流水线搭建与维护 |
| **最大痛点** | 团队 Jenkins + Vector + Jira + Polarion 各跑各的，追溯链断裂，集成花的时间比开发还多 |
| **预算权限** | ✅ 有技术预算 + 架构决策权 |
| **决策风格** | 技术理性——看重架构清晰度、扩展性、可维护性 |

### 价值主张

> **"工具链不能变成另一座孤岛。"**

yuleOSH 为架构师解决的核心问题：
1. **Pipeline 编排层** — 不替代现有工具，而是做 CI/CD + 合规 + 追溯的编排中枢
2. **开箱适配器** — OpenOCD/JLink/esptool 硬件适配 + GitLab/GitHub/Jenkins/Polarion 集成
3. **Vector/dSPACE 生态寄生** — 读/写 CANoe 回放数据、兼容 vTESTstudio 测试格式
4. **插件架构** — 自定义 adapter，快速对接内部 Legacy 工具链

### 典型场景

> 刘博正在评估新项目的工具链方案：团队用 Jenkins 做 CI，Jira 管需求，Polarion 做 ALM。传统方案需要在三者之间写自定义脚本同步数据。yuleOSH 的 OpenSpec 引擎直接读 Jira 需求 → 自动触发 Pipeline → 证据回写 Polarion。**集成工作从 4 周压缩到 1 天。**

## 画像 3：开发者（QEMU · CI）

### 人物画像

| 维度 | 描述 |
|:-----|:------|
| **姓名示例** | 小张，嵌入式 C 开发工程师（3 年经验） |
| **团队规模** | 10-15 人嵌入式开发团队 |
| **日常工作** | 写 C 代码、跑测试、修 Bug、被 Code Review 阻塞 |
| **最大痛点** | 每次提交要等 30 分钟 CI → 等同事 Code Review → 再出新版本 → 再 CI。**每天真正写代码的时间不到 4 小时** |
| **预算权限** | ❌ 无预算权限，但可向上推荐 |
| **决策风格** | 实用驱动——"能不能让我早点下班" |

### 价值主张

> **"从需求到测试，AI 替你走完 80% 的路。"**

yuleOSH 为开发者解决的核心问题：
1. **AI Agent 全自动管线** — 写入需求 → AI 自动完成 SDD→DDD→Code→Test→Review，开发者只需验收
2. **QEMU SIL 快速验证** — 无需硬件板子，本地 QEMU 仿真跑完整 Pipeline，90 秒看结果
3. **三层 CI/CD 不阻塞** — Layer 1 单元+覆盖率 → Layer 2 交叉编译+静态分析 → Layer 3 系统验证，并行跑
4. **一句命令安装** — `pip install yuleosh && yuleosh init project`，两分钟上线

### 典型场景

> 小张接到一个 UART 驱动的需求。以前：写需求文档 → 等评审 → 写代码 → 等 CR → 跑测试 → 出 Bug → 改 → 再 CR → 合入。**一个简单的 UART 驱动要走 3 天。** 现在：在 yuleOSH 写 10 行 spec → 回车 → AI Agent 自动完成设计/编码/测试/审查 → 小张检查产出物 → approve → 合入。**全程 2 小时。**
