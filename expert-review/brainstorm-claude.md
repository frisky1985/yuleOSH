## 2. 三条路线分析

### 路线 A: 产品化冲刺

**核心工作：** SaaS 底座（多租户/SSO/用量计量）、Onboarding Wizard、Landing Page 部署上线

**技术难点：**

| 难点 | 严重程度 | 分析 |
|------|:--------:|------|
| 多租户架构设计 | 🔴 高 | 当前 store 模块的数据模型没有租户隔离字段（tenant_id），全部表需要加字段 + migration + 全部查询加 WHERE tenant_id = ?。需要评估是按 DB schema 隔离还是 row-level 隔离 |
| SSO 集成 (OIDC/SAML) | 🟡 中 | Go 生态有成熟库（dex、casdoor），但需要引入认证中间件、session 管理、RBAC 模型。估算 5-7 天 |
| 用量计量 | 🟡 中 | 需要设计计量模型（按 CI 调用次数 / 知识条目数 / 存储量），接入计费网关 |
| Landing Page + Onboarding | 🟢 低 | 纯前端工作，技术风险低 |
| SaaS 部署运维 | 🔴 高 | 当前代码是单机可执行，要做 SaaS 需要：容器化（Dockerfile 已有？）、k8s manifest、数据库连接池优化、日志聚合。**当前完全空白** |

**工时估算：**

| 工作项 | 人·天 | 风险系数 |
|--------|:-----:|:--------:|
| 多租户基础设施（字段 + migration + 查询改造） | 8-10 | 高 |
| SSO 集成（OIDC + RBAC） | 5-7 | 中 |
| 用量计量模块 | 3-5 | 中 |
| Landing Page + Onboarding Wizard | 5-7 | 低 |
| SaaS 容器化 + k8s 部署 | 5-8 | 高 |
| **合计** | **26-37** | |

**优先级排序（如果做路线 A）：**
1. **先做 Landing Page**（2-3 天可上线，给老板看到"产品感"）
2. **SSO 集成**（没有登录没法用）
3. **SaaS 容器化部署**（先单租户上线，再改多租户）
4. **多租户改造**（等有 2+ 客户再投入）
5. **用量计量**（付费才需要，最次优先级）

### 路线 B: 市场验证

**核心工作：** Fix 已知 bug、完善文档、写教程、做个漂亮的 landing page、找客户试用

**技术难点：**

| 难点 | 分析 |
|------|------|
| 已知 bug — 不多 | Phase 1 交付中 bug 不多，测试覆盖较好。风险较低 |
| 文档深度 | 当前有 Spec 文档但缺乏：API 文档（OpenAPI 未生成）、用户教程、部署指南 |
| Landing Page | 纯前端，技术风险低 |
| "找客户试用"的准备工作 | **最大的隐性工作量：** 产品需要能跑通一个真实端到端场景（从 CI 集成 → 知识存储 → FMEA 分析 → 报告生成）。当前端到端验证尚未完成 |

**产品 ready 了吗？** → **严格来说：NO**

具体来说：

```
✅ 能做：
  - MISRA cppcheck 解析 + 报告
  - pytest JUnit XML 解析 + 报告
  - Pipeline 编排器（通用化）
  
❌ 不能做（缺的致命能力）：
  - 命令行工具 (CLI) — 用户怎么启动？没有 yuleosh CLI，只有 Python 脚本
  - 知识管理 — 全在 Spec 阶段，0 行代码
  - FMEA — 0 行代码
  - API 服务器 — Go service 在设计阶段，没有可启动的 HTTP 服务
  - Web UI — 没有
  - 安装方式 — 没有 pip install 或 docker pull
```

**核心矛盾：** 
- 想卖的产品应该是**知识管理闭环工具**（KB + FMEA + LL + CI 集成）
- 实际上是 **CI 报告生成 Python 库** + **一叠很厚的 Spec 文档**

**评价：** 路线 B 在当前阶段过早。除非我们愿意定位为 "MISRA 报告 CLI 工具" 且目标用户是极客工程师，否则需要先走一段路线 C 把核心能力补齐。

### 路线 C: 深度攻坚

**核心工作：** 覆盖率全局 30%+、ISO 26262 功能安全初版（ASIL 标注）、AUTOSAR Phase 3（BSW 配置助手）、实时性分析

**技术难点：**

| 难点 | 严重程度 | 分析 |
|------|:--------:|------|
| 覆盖率分析引擎 | 🟡 中 | 需要集成 gcov/lcov 或 LLVM Source-based Coverage，聚合到统一视图。技术可行，估算 5-7 天 |
| ASIL 标注引擎 | 🔴 高 | 这不是简单的字段，需要：代码级 ASIL 标记（函数/变量）、ASIL 分解规则（ASIL D → ASIL B(D)+ASIL B(D)）、独立性分析（freedom from interference — FFI）。这是 ISO 26262-6 的核心要求 |
| AUTOSAR BSW 配置助手 | 🔴 高 | BSW 配置是 AUTOSAR 里最复杂的部分之一，涉及 CAN/LIN/Ethernet 协议栈、OS、Memory Stack 等。做一个有实际价值的配置助手需要深厚的 AUTOSAR 知识 |
| 实时性分析 (WCET) | 🔴 高 | 需要静态分析（AI-based 或 abstract interpretation）或测量方法（RTA/Trace），工具链复杂 |

**对产品价值提升有多大？**

| 工作项 | 价值提升 | 理由 |
|--------|:--------:|------|
| 覆盖率 30%+ | ★★★★☆ | 直接对客户有量化展示，"我们的工具帮你的工程从 5% 提升到 30%" 是可以写在官网数据 |
| ISO 26262 ASIL 标注 | ★★★★★ | 这是功能安全工具链的核心能力。能标注 ASIL = 能打"功能安全工具"标签 = 能卖高价（对比：MISRA 工具遍地都是） |
| AUTOSAR BSW 配置助手 | ★★★☆☆ | 价值很大但实施成本极高，且竞品（Vector DaVinci、EB tresos）很成熟 |
| WCET 分析 | ★★★★★ | 高性能嵌入式实时系统刚需，但准确 WCET 分析工具需要大量实证验证 |

**评价：** 路线 C 对产品价值提升最大，但单点突破比全面开花更明智。

## 3. 如果你有一个月时间，只能做一件事？

### 我的选择：**实现端到端 MVP 闭环 — 从 CI 触发到知识存储**

**具体来说，一个月时间把这条路打通：**

```
开发者 A: git push
    │
    ▼
CI (Pre-commit Hook): 跑 MISRA cppcheck ✅ (已有)
    │
    ▼
CI (Post-merge): 解析 cppcheck 报告 → 自动检查是否有新的违规
    │
    ▼
知识存储: 新的代码变更自动关联到 KbArticle / LL ✅ (Phase1 需实现)
    │
    ▼
Web UI (最小): 展示 MISRA 违规趋势图 + 知识库浏览
```

**工作量分解（30 天）：**

| 周 | 工作项 | 交付物 |
|:--:|--------|--------|
| 第 1 周 | **Go 知识管理服务骨架**：DB migration 跑通、KB CRUD API、最小 store 实现 | `POST /api/v1/kb/articles` 能创建和查询 |
| 第 2 周 | **CI Pipeline 集成**：Pre-commit hook 调用 yuleosh CLI、Post-merge 自动创建 KB 条目、MISRA 违规自动捕获 | CI 跑完 → 知识库自动生成"违规报告"条目 |
| 第 3 周 | **最小 Web UI**：知识库浏览列表 + MISRA 趋势图表 + 搜索 | 用户打开浏览器能看到东西 |
| 第 4 周 | **回填 + 打磨**：bug fix、用真实项目数据回填（跑一次完整流程）、写部署文档（docker compose up 就能用） | 能给第一个人演示完整闭环 |

### 为什么选这个？

**理由 1：解决"漂亮的半成品"问题。**
当前产品的问题不是"功能不够多"，而是**没有一个能跑的端到端 demo**。知识管理的 Spec 写了 40 页，但用户点开看到的是 0 行代码。

**理由 2：最高 ROI**
- 不需要等 IS026262（需要 2-3 个月才能出能用的初版）
- 不需要等 AUTOSAR（需要 3+ 个月，且竞品太强）
- 不需要等多租户/SSO（做了也没人用，因为产品本身还没闭环）

**理由 3：为两条路线都留后路**
- 这个 MVP 做完了 → 再加多租户和 SSO 就是路线 A
- 这个 MVP 做完了 → 拿着它给客户演示就是路线 B
- 这个 MVP 做完了 → 再往上加 ASIL 标注就是路线 C

**总结：打通"CI push → 知识库自动生成"这个 0→1 的闭环，比任何单点深度都有价值。**

## 4. 紧急 Bug / 风险

### 🔴 P0 — 必须修

| # | 风险 | 粒度 | 影响 | 修复要求 |
|:-:|------|:----:|------|----------|
| 1 | **Mock 测试过多，真实集成测试太少** | 架构 | 所有 pipeline 测试都 mock 了 subprocess。一旦真实 cppcheck 输出格式有变化（跨版本），测试全部通过但生产环境崩 | 至少加 2 个集成测试（真实调用 cppcheck/pytest），不做全量但验证主要流程 |
| 2 | **知识管理 Spec 过度设计** | 架构 | 当前 Spec 设计了 16 张表 + 审计系统 + OSLC Provider + Polarion 同步 + ReqIF。Phase 1 48 人·天，但项目实际人力 1-2 人。**这个范围在 1-2 人的小团队不可能交付** | 需要砍到真正 MVP：3 张表（kb_articles, lessons, fmea_entries）+ 简单 CRUD API。其他全部丢到"Phase 2+ later" |

### 🟡 P1 — 建议尽快修

| # | 风险 | 分析 |
|:-:|------|------|
| 3 | **测试中有大量空函数桩** | 从 Pipeline 框架的 38 个测试来看，不少测试测的是"函数返回了默认值"。这些测试会在重构时碎一地，而且给了"我们有测试"的虚假安全感 |
| 4 | **Schema 锁死过早** | 早期的 FMEA/FMEA items 设计了 generated column (`rpn = severity * occurence * detection`)，但现在还没有 RPN 计算引擎。设计跑在前面，代码跟不上会导致"先改 DB schema 再改代码"的 reverse work |

### 🟢 P2 — 注意但不紧急

| # | 风险 | 分析 |
|:-:|------|------|
| 5 | **没有 CLI 入口** | CI 集成需要 CLI，但当前所有功能在 Python 脚本里。用户没法 `yuleosh ci-analyze --project-dir .` |
| 6 | **Python 版本依赖未锁定** | 在 Mac 上开发但有 ARM Linux 部署需求（嵌入式 CI runner 通常是 Linux）。`requirements.txt` 或 `pyproject.toml` 未看到，存在架构兼容风险 |
