| 建议2（证据包修复） | 🟢 低（功能补全） | 🟢 低 | 🟢 低 | 🟢 **低** |
| 建议3（C覆盖率） | 🟡 中（代码可测试性差） | 🟡 中（15人天=小克主力3周） | 🟢 低 | 🟡 **中** |

### 4.2 详细风险分析与缓解措施

**风险 R3-1: C 覆盖率攻坚遇到不可测试代码**
- **描述**: 部分核心模块（如 `ci/kpi/stability.py` KPI 稳定性分析）可能深度耦合外部系统，导致 unit testing 非常困难
- **后果**: 覆盖率无法达到目标，时间超出 Sprint 1 预算
- **缓解措施**:
  1. 先做"最低挂果"：从 `evidence/` 和 `flash/` 等模块性较强的入手，积累 momentum
  2. 如果某模块重构成本过高，改为"集成测试覆盖"（用 CI 端到端测试替代纯单元测试）
  3. 允许模块级别差异化目标：简单模块 ≥70%，困难模块 ≥40%，整体 ≥30%
- **延期影响**: Sprint 1 如果未达 15%，Sprint 2 的压力会显著增大。最坏情况：Sprint 2 中期达 15% → 20%，Sprint 3 达 30%。**不建议压缩时间**——覆盖率冲刺靠数量不靠速度，赶出来的测试质量低反而会被审计师识破

**风险 R4-1: Benchmark 任务集代表性受质疑**
- **描述**: 如果 50 个任务集中在简单/中等场景，行业专家可能会质疑"挑任务"的 bias
- **后果**: Benchmark 公开后受到第三方质疑，反而损害产品信誉
- **缓解措施**:
  1. 按难度等级（简单/中等/困难）分层设计任务集
  2. 在报告中公开每个任务的详细描述和难度评级
  3. 邀请行业专家评审任务集（可以在 Sprint 2 做一次小范围的 peer review）
- **延期影响**: 如果 Sprint 1 未完成 Benchmark 数据采集，Sprint 2 的公开时间将推迟。但不影响其他任务的执行

**风险 R5-1: MISRA 官方规则文本版权限制**
- **描述**: MISRA C:2012 官方规则文本受版权保护，不能直接全文索引到 RAG
- **后果**: RAG 索引只能包含"摘要"或"规则编号+参考"，效果打折扣
- **缓解措施**:
  1. RAG 索引使用公开的 MISRA 文档摘要（非全文）
  2. 开发"规则查询适配器"——不是直接索引全文，而是索引 yuleOSH 自身的偏差管理记录和常见违规模式
  3. 法律确认：如果确实不能使用，改为 Agent 的 few-shot prompt 方式替代 RAG
- **延期影响**: 低。即使 MISRA RAG 不完整，嵌入式最佳实践 RAG 和 历史审查 RAG 仍然有价值。Sprint 1 内可以完成替代方案

**风险 R6-1: AUTOSAR ARXML 解析远超预期复杂度**
- **描述**: ARXML 使用 AUTOSAR XSD Schema，复杂度极高。3 天实现的解析器可能只能覆盖 60% 的常见场景
- **后果**: Phase 1 在 Sprint 3 结束时无法交付 80% 以上的 ARXML 兼容性
- **缓解措施**:
  1. 明确定义 Phase 1 的"够用"标准——只解析 SWC 层级（Component / Port / Runnable），不解析完整的内核配置
  2. 从最简单的主流 ARXML 变体（Vector DaVinci 导出）开始做兼容，后续逐步扩展
  3. 如果时间不够，Phase 1 拆分到 Sprint 3 + Sprint 4（延期但不影响其他 P0）
- **延期影响**: AUTOSAR 对接是 P1 而非 P0，延期不会影响核心质量整改。但如果 Sprint 3 完全未交付 ARXML 解析，Sprint 4（若有）需要追加 3-5 天

**风险 R7-1: Dashboard 上线时覆盖率数据仍偏低**
- **描述**: 如果建议 3 的覆盖率攻坚在 Sprint 2 结束时只有 20%，质量经理 Dashboard 显示"覆盖率 20%"会让用户不信任
- **后果**: Dashboard 变成"展示丑陋数字"的平台
- **缓解措施**:
  1. Dashboard 显示"趋势"而非"绝对值"——展示覆盖率从 1.4%→20% 的提升曲线，传递"我们正在认真改进"的信号
  2. 在覆盖率 <30% 的模块旁标注"攻坚中"，避免用户误以为是最终状态
  3. 如果 Sprint 2 结束时覆盖率仍 <15%，推迟 Dashboard 上线至 Sprint 3

## 8. 总结

### 核心结论

**yuleOSH v1.0.0 GA 的方向正确，但产品成熟度不足。** 综合 6.1/10 的评分说明了 gap 所在。

**必须在 6 周（3 个 Sprint）内解决的问题优先级**:

1. **立即（第 1 周）**: 定位调整 + 证据包修复。这两个不依赖其他模块，改了就能改善产品 credibility
2. **攻坚（第 1-4 周）**: C 覆盖率 1.4%→30%。这是硬骨头，也是评分提升的关键。覆盖率从 1.4% 到 30% 的 climb 比从 50% 到 70% 更有叙事价值——因为从"几乎没测"到"有系统地在测"是质变
3. **持续（第 1-6 周）**: AI Benchmark + LLM/RAG 策略。这两项补全了 AI 工具行业对"AI 原生"产品的数据期待和架构透明度
4. **深化（第 3-6 周）**: 质量经理 Onboarding + AUTOSAR + 场景故事。这些是产品化的"最后一公里"
