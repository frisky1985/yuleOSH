## 功能 F-01: 合规概览仪表盘

### 功能名
SWE.1~SWE.6 合规状态仪表盘

### 用户故事
> **作为一个** 质量经理，
> **我希望** 登录后默认看到当前项目在 ASPICE SWE.1~SWE.6 每个过程域的完成状态，
> **以便** 快速了解合规进度，知道"哪里完成了、哪里还有 gap"。

### 验收标准

| # | 验收项 | 判定方式 |
|:-:|:-------|:---------|
| F-01.1 | 登录后默认跳转到 Dashboard 首页，URL 路径为 `/dashboard` | 浏览器导航验证 |
| F-01.2 | Dashboard 展示 SWE.1~SWE.6 共 6 个过程域卡片或行，每行含：过程域名称、当前状态（✅ 完成 / ⚠️ 部分 / ❌ 未开始） | 视觉检查 + 状态标签存在 |
| F-01.3 | 每个状态标签的颜色语义正确：✅ 绿色 / ⚠️ 黄色 / ❌ 红色 | 视觉检查 |
| F-01.4 | 每个过程域卡片包含"详情"链接，点击后跳转到当前过程域的证据详情页 | 链接可点击，目标 URL 存在 |
| F-01.5 | 仪表盘顶部显示整体合规进度百分比（已完成过程域数 / 6 × 100%） | 数值计算验证 |
| F-01.6 | 仪表盘数据来源为证据包中的 `audit-manifest.json`（建议 2 产出），非硬编码假数据 | 修改 manifest 后仪表盘状态变化 |
| F-01.7 | 如果证据包不存在或解析失败，仪表盘显示友好提示："合规数据尚未生成，请先运行 <yuleosh ev pack> 或点击"生成证据包"按钮" | 空状态验证 |
| F-01.8 | 覆盖率数据（行覆盖、分支覆盖）在仪表盘侧边或底部区域以数字或迷你图表展示 | 数值存在且大于 0 |

## 功能 F-02: 项目清单

### 功能名
多项目管理与切换

### 用户故事
> **作为一个** 质量经理，
> **我希望** 看到我负责的所有项目列表，
> **以便** 在不同项目之间快速切换，查看各项目的合规状态。

### 验收标准

| # | 验收项 | 判定方式 |
|:-:|:-------|:---------|
| F-02.1 | 仪表盘左侧或顶部导航区域显示项目列表下拉菜单或侧边栏 | 视觉确认 |
| F-02.2 | 项目列表从后端 API 获取（支持至少 1 个默认项目），项目名取自 yuleOSH 项目配置 | API 响应验证 |
| F-02.3 | 切换项目后，仪表盘内容（SWE 状态、覆盖率、差距分析）全部刷新为选中项目的数据 | 切换后数据变更 |
| F-02.4 | 每个项目卡片/行显示：项目名称、最后更新时间、SWE 完成数（如 "4/6"） | 视觉确认，字段存在 |
| F-02.5 | 支持项目搜索过滤（≥3 字符键盘输入过滤），搜索结果实时更新 | 输入 "co" 显示包含 "core" 的项目 |
| F-02.6 | 支持"新建项目"入口（MVP 阶段可仅跳转到 CLI 指引页，无需完整创建流程） | 按钮存在，点击后显示指引信息 |
| F-02.7 | 当前选中项目高亮标识 | 视觉确认（背景色或边框） |

## 功能 F-03: 证据包一键生成

### 功能名
One-Click Evidence Pack 生成

### 用户故事
> **作为一个** 质量经理，
> **我希望** 在 Dashboard 上点击一个按钮就能触发证据包生成，
> **以便** 不需要记住 CLI 命令（如 `yuleosh ev pack`），降低使用门槛。

### 验收标准

| # | 验收项 | 判定方式 |
|:-:|:-------|:---------|
| F-03.1 | Dashboard 上存在显眼的"生成证据包"按钮（页面右上角或 SWE 模块顶部） | 视觉确认 |
| F-03.2 | 点击按钮后触发后端证据包生成流程（调用 `yuleosh ev pack` 或等效 API） | 日志验证后端命令被执行 |
| F-03.3 | 生成过程中显示进度指示器（进度条或旋转器 + "正在生成证据包..."文案） | 视觉确认 |
| F-03.4 | 生成完成后结果显示：✅ 成功，并显示下载链接（ZIP 文件路径或直接下载）或 ❌ 失败 + 错误信息 | 成功/失败两种状态验证 |
| F-03.5 | 生成结果的 `valid` 字段必须为 `True`（依赖建议 2 修复） | 后端 `yuleosh ev check` 返回 `valid: True` |
| F-03.6 | 生成的证据包包含 `audit-manifest.json` + SWE 各过程域证据文件 + `llm-call-log.json`（建议 5 产出，Sprint 2 末） | ZIP 内容验证 |
| F-03.7 | 生成完成后，Dashboard 视图自动刷新，SWE 状态仪表盘更新为最新数据 | 数据自动刷新 |
| F-03.8 | 按钮文本在生成过程中不可重复点击（disabled 状态），完成后恢复可用 | 点击后按钮 disabled 样式确认 |

**注意**: F-03 强依赖建议 2（证据包修复）和部分建议 3（覆盖率数据）。如果覆盖率仍然 < 5%，生成的证据包应附带 WARNING 提示（建议 2.5 要求），不影响生成功能本身。

## 功能 F-04: 差距分析视图

### 功能名
SWE 合规差距分析

### 用户故事
> **作为一个** 质量经理，
> **我希望** 看到当前项目与 ASPICE SWE 完整要求之间的差距清单，
> **以便** 明确知道"还需要做哪些工作"才能通过审计。

### 验收标准

| # | 验收项 | 判定方式 |
|:-:|:-------|:---------|
| F-04.1 | Dashboard 中存在"差距分析"标签页或独立面板，可从导航进入 | 视觉确认，URL 路径为 `/dashboard/gap-analysis` |
| F-04.2 | 差距分析以表格形式展示每个 SWE 过程域的差距条目 | 表格存在，至少包含以下列：过程域、差距描述、严重级别（Critical/Major/Minor）、当前状态 |
| F-04.3 | 严重级别使用颜色标识：🔴 Critical / 🟡 Major / 🟢 Minor | 颜色视觉确认 |
| F-04.4 | 差距分析数据来源为 yuleOSH 的合规检查结果（证据包中的差距项），非假数据 | 修改证据包差距项后视图变化 |
| F-04.5 | 差距条目可折叠/展开（Show 10 more），默认显示最近 10 条 | 点击"Show 10 more"显示更多 |
| F-04.6 | 每个差距条目如果已知解决方案或修复建议，应显示"建议"列 | 列存在，内容非空 |
| F-04.7 | 差距分析顶部显示汇总数字："共 N 项差距，其中 Critical M 项 / Major K 项 / Minor L 项" | 汇总数字计算正确 |
| F-04.8 | 提供"导出差距报告"按钮，可将当前差距分析导出为 CSV 或 PDF | 按钮存在，导出文件格式正确 |

## 接口契约（概要）

### Dashboard 后端 API 端点（MVP 阶段）

| 端点 | 方法 | 输入 | 输出 | 对应功能 |
|:-----|:-----|:-----|:------|:---------|
| `/api/v1/dashboard/projects` | GET | — | `[{id, name, last_updated, swe_completed_count, swe_total}]` | F-02 |
| `/api/v1/dashboard/swe-status` | GET | `?project_id=` | `{swe: {SWE1:{status, details_url}, ...}, overall_pct}` | F-01 |
| `/api/v1/dashboard/gap-analysis` | GET | `?project_id=&page=&limit=` | `{items:[{swe_area, description, severity, status, suggestion}], summary:{total, critical, major, minor}}` | F-04 |
| `/api/v1/dashboard/evidence/generate` | POST | `{project_id}` | `{task_id, status}`（返回后轮询状态） | F-03 |
| `/api/v1/dashboard/evidence/status` | GET | `?task_id=` | `{status, progress_pct, download_url, valid}` | F-03 |
| `/api/v1/dashboard/coverage` | GET | `?project_id=` | `{line_pct, branch_pct, trend:[{date, line_pct}]}` | F-01/F-03 |

### 数据依赖关系

```
F-01 合规仪表盘 ──── 需要建议 2（证据包 valid）───→ audit-manifest.json
F-01 覆盖率数据  ──── 需要建议 3（≥15% 覆盖率） ──→ coverage.json
F-03 证据包生成  ──── 需要建议 2（ev pack CLI） ───→ yuleosh ev pack 命令正常
F-03 生成后刷新  ──── 需要 F-01 数据就绪  ──────→ 循环依赖？不——刷新触发新 API 调用
F-04 差距分析    ──── 需要建议 3（合规检查） ────→ gap-analysis.json
F-06 3 步向导    ──── 需要 F-04 就绪 + 建议 7.3（文档上传解析）
```

## 开发约束

1. **MVP 的假数据策略**: 如果某功能的数据源（建议 2/3）未完全就绪，允许使用假数据做 UI 开发，但必须在数据区域标注 **"⚠️ 演示数据 — 需连接实际项目"** 的提示条
2. **覆盖率低值的显示策略**: 如果覆盖率 < 30%，不显示绝对值的大号数字，改为显示"覆盖率提升趋势"（从 1.4%→当前值的 U 型曲线），避免用户对低覆盖率产生信任危机
3. **性能要求**: Dashboard 首页首屏加载 ≤ 3 秒（API 响应 ≤ 1s，渲染 ≤ 2s）
4. **浏览器兼容**: 支持 Chrome 120+ / Firefox 120+ / Safari 17+（仅桌面端）
